# 프로젝트 현재 상태와 다음 작업

이 문서는 다른 AI 에이전트나 개발자가 프로젝트를 이어받을 때 가장 먼저 확인하는 작업 로드맵입니다.

## 현재 요약

MES/MCS 기본 업무와 React 전환 1차 작업은 구현되어 있습니다. 현재 프로젝트의 핵심 시연 흐름은 `MES 작업오더 -> MCS 자재 이동 요청 -> MCS 경로 계산 -> PLC 이벤트 -> MES 작업 시작 허용/차단`입니다.

## 완료된 작업

| 영역 | 상태 | 비고 |
|---|---|---|
| MES 기본 기능 | 완료 | 생산계획, 작업오더, 기준정보, 생산 흐름 기반 |
| MCS 기본 기능 | 완료 | Zone, Location, 로케이션 재고, 입고, 출고, 이동 관리 |
| React 전환 | 진행 완료 | MES/MCS 주요 화면을 React에서 사용 |
| MES/MCS DB 통합 | 완료 | 동일 `MES_DB` 사용, MCS 테이블은 `MCS_` 접두어 |
| SQL/문서 정리 | 완료 | `docs/README.md`, `docs/db/DB_FILES.md` 기준 |
| MCS 이동 관리 | 완료 | 이동오더/이동품목/재고 반영/이력 처리 |
| MCS 경로 최적화 | 완료 | 노드/구간을 DB에 저장하고 그래프처럼 계산 |
| PLC 시뮬레이터 | 완료 | 성공, 실패, 복구 시나리오용 PowerShell 스크립트 |
| MES 자재 요청 연동 | 완료 | MES 작업오더에서 MCS 이동오더 생성 |
| 작업 시작 인터락 | 완료 | MCS 자재 이동 완료 전 MES 작업 시작 차단 |
| 실패 후 복구 흐름 | 완료 | 실패 이동 건 취소 후 재요청 가능 |

## 현재 기준 시연 흐름

```text
1. MES에서 작업오더를 생성한다.
2. 작업오더에서 자재 요청을 누른다.
3. MCS가 재고 LOT, 출발/도착 Location, 경로 기준을 자동 배정해 이동오더를 생성한다.
4. MCS에서 이동오더를 시작하거나 PLC 시뮬레이터로 시작 이벤트를 보낸다.
5. 성공 시 MCS 이동오더가 완료되고 MES 작업 시작이 가능해진다.
6. 실패 시 MCS 이동오더가 실패 상태가 되고 MES 작업 시작은 차단된다.
7. 실패 건은 취소 후 다시 자재 요청해서 복구 시나리오를 시연한다.
```

## 다음 개발 순서

### 1. AI 운영 분석 기반 만들기

목표는 AI가 DB를 직접 변경하지 않고, 현재 운영 상태를 읽어서 문제 원인과 조치 방향을 설명하는 것입니다.

우선 구현 범위:

- 작업오더/MCS 이동오더/PLC 이벤트/재고 상태를 읽는 운영 스냅샷 API
- Spring AI를 호출하는 backend 서비스
- “현재 문제 요약”, “원인 추정”, “다음 조치” 형태의 분석 응답
- React 대시보드 또는 작업오더 화면에 AI 분석 패널 표시

참조 문서: [design/AI_OPERATION_PLAN.md](design/AI_OPERATION_PLAN.md)

### 2. 자연어 운영 조회

사용자가 챗봇처럼 질문하면 MES/MCS 상태를 조회하고 설명합니다.

예시 질문:

- “지금 시작 못 하는 작업오더가 뭐야?”
- “MCS 이동 실패 건이 왜 발생했어?”
- “NCM811 자재 이동 현황 보여줘.”

### 3. AI 이벤트 알림

실패, 지연, 인터락, 재고 부족 같은 이벤트가 생기면 AI가 운영자가 이해하기 쉬운 알림 문장을 만듭니다.

초기 알림 채널:

- React 알림 목록
- 이메일 또는 문자 연동은 후순위

## 설계 원칙

- MES와 MCS는 별도 시스템처럼 역할을 분리합니다.
- MES는 “무엇을 생산할지/작업할지”를 관리합니다.
- MCS는 “어떤 자재를 어느 경로로 이동할지”를 관리합니다.
- MES는 자재 요청만 하고, 출발지/LOT/경로 상세 선택은 MCS가 담당합니다.
- AI는 상태 변경 주체가 아니라 분석/설명/알림 보조 역할입니다.
